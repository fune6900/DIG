検閲のメイド（QAサブエージェント）を起動し、Playwright MCPを使用してE2Eテストを実行する。以下の手順を厳守すること。

## 対象
- 引数 $ARGUMENTS にテスト対象のURL、またはフロー名を指定する。
- 未指定の場合は `http://localhost:3000` を起点に全主要フローをテストする。
- フロー名指定例: `knowledge`（ナレッジ検索）、`tryon`（AI試着）、`diary`（コーデ日記）

## 手順

### Phase 0: 事前確認
- `mcp__playwright__browser_navigate` で対象URLへ遷移し、ページが正常に表示されることを確認する。
- `mcp__playwright__browser_console_messages` でページロード時のエラーを確認する。
- エラーがある場合はテスト続行前にマスターへ報告すること。

### Phase 1: ナレッジ検索フロー
（`knowledge` 指定時、または全テスト時に実行）

1. 検索フォームの存在確認（`mcp__playwright__browser_snapshot`）
2. `mcp__playwright__browser_type` で検索キーワードを入力（例: "リーバイス 501 年代"）
3. `mcp__playwright__browser_press_key` でEnterまたは送信ボタンをクリック
4. 検索結果の表示を確認（`mcp__playwright__browser_take_screenshot`）
5. 検索結果が0件の場合のエンプティステート表示を確認
6. 不正な入力（空文字、特殊文字、超長文字列）での挙動を確認

**検証項目**:
- [ ] 検索フォームが表示されている
- [ ] キーワード入力が反映される
- [ ] 送信後に結果が表示される（もしくはローディングが表示される）
- [ ] 0件時にエンプティステートが表示される
- [ ] 不正入力でエラーになってもクラッシュしない

### Phase 2: AI試着フロー
（`tryon` 指定時、または全テスト時に実行）

1. 商品一覧または商品詳細ページへ遷移
2. 試着ボタンの存在確認
3. `mcp__playwright__browser_click` で試着ボタンをクリック
4. ローディング状態の表示を確認（`mcp__playwright__browser_take_screenshot`）
5. 試着結果の表示を確認
6. 試着キャンセル・再試着の動作を確認

**検証項目**:
- [ ] 試着ボタンが表示されている
- [ ] クリック後にローディングが表示される
- [ ] 結果が表示される
- [ ] キャンセル操作が機能する
- [ ] 連続クリックで二重送信が発生しない（ボタンがdisabledになるか確認）

### Phase 3: コーデ日記フロー
（`diary` 指定時、または全テスト時に実行）

1. 日記一覧ページへ遷移
2. 新規投稿ボタンの存在確認
3. `mcp__playwright__browser_click` で投稿ボタンをクリック
4. フォーム表示の確認（`mcp__playwright__browser_snapshot`）
5. `mcp__playwright__browser_fill_form` で必須項目を入力
6. 保存ボタンをクリックし、完了状態を確認
7. バリデーション確認（必須項目未入力で送信）
8. 投稿した記録が一覧に反映されることを確認

**検証項目**:
- [ ] 投稿ボタンが表示されている
- [ ] フォームが開く
- [ ] 必須項目を入力して保存できる
- [ ] 未入力の場合にバリデーションエラーが表示される
- [ ] 保存後に一覧へ反映される

### Phase 4: 共通UX検証
全フローで以下を確認する。

1. **ナビゲーション**: `mcp__playwright__browser_navigate_back` でブラウザバックが正常に機能するか
2. **ネットワークエラー**: `mcp__playwright__browser_evaluate` でオフライン状態をシミュレートし、エラーハンドリングを確認
3. **タブ管理**: `mcp__playwright__browser_tabs` で意図しないタブ開きが発生していないか確認
4. **最終スクリーンショット**: 各フロー完了時に `mcp__playwright__browser_take_screenshot` でキャプチャ

### Phase 5: 回帰確認
既存テストファイルが存在する場合は `Bash` で `npm run e2e` を実行し、既存テストとの整合性を確認する。
新規のテストケースが発見された場合は `Write` で `tests/e2e/` 配下にPlaywrightテストコードとして追記する。

## 注意
- 開発サーバーが起動していない場合は `npm run dev` の実行をマスターに促すこと。
- FAILの原因を「不明」で終わらせるな。`mcp__playwright__browser_evaluate` でDOM状態やJSエラーを掘り下げ、根本原因まで特定すること。
- 応答時間が500ms超のステップは「要改善」として別途フラグを立てること。
- テストが意図せずデータを変更した場合（投稿・削除等）は、テスト後にクリーンアップを行うかマスターへ報告すること。
- 新規テストコードは既存の `tests/e2e/` の命名規則・構造に従うこと。ない場合は `<flow-name>.spec.ts` 形式で作成する。

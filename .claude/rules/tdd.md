# TDD ルール

## サイクル

### Red（失敗するテストを書く）
- **検閲のメイド（QA）** が担当
- 実装コードより先にテストを書く。これは義務
- テストは「仕様の文書」。何をすべきかを明示する
- `npm test` を実行し、テストが失敗することを確認してから次へ進む

### Green（テストを通す最小限のコードを書く）
- **構築のメイド（Coder）** が担当
- テストをパスさせることだけを考える
- 綺麗さは後回し。まず動かす
- `npm test` が全件グリーンになるまで実装を続ける

### Refactor（リファクタリング）
- **メイド長（Benz）** が監督
- テストがグリーンのまま、コードの品質を上げる
- 重複の排除、命名の改善、構造の整理
- リファクタリング後も `npm test` がグリーンであることを確認

---

## テスト種別と担当ツール

| 種別 | ツール | 対象 | コマンド |
|------|--------|------|---------|
| ユニットテスト | Vitest + React Testing Library | コンポーネント・関数・hooks | `npm test` |
| E2Eテスト | Playwright | ユーザーフロー全体 | `npm run e2e` |
| 視覚的テスト | `/visual-regression` | レイアウト・スタイル | `/visual-regression` |

---

## 鉄則

- **No Test, No Code**: テストのないコードはレビュー対象外。`/review-pr` で弾く
- **テストファイルの場所**: `tests/unit/` にユニット、`tests/e2e/` にE2E
- **テストの命名**: `describe('ComponentName')` > `it('should <期待する挙動>')`
- **モックの制限**: DBのモックは禁止（実際のDBを使う）。外部API・時刻のみモック許可
- **カバレッジ**: 新規コードは必ずテストを書く。カバレッジの数値目標より「重要なパスが通っているか」を優先する

---

## エージェント呼び出し順序

```
1. QA（検閲のメイド）→ テスト設計・Red フェーズ
2. Architect（礎のメイド）→ 型定義・スキーマ定義
3. Coder（構築のメイド）→ Green フェーズ（実装）
4. Designer（図案のメイド）→ UIコンポーネント（必要な場合）
5. Benz（メイド長）→ Refactor 監督・最終確認
```
